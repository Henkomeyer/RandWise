## Completed
- Implemented real `/api/v1/auth/register`, `/api/v1/auth/login`, `/api/v1/auth/refresh`, `/api/v1/auth/logout`, and `/api/v1/auth/me` behavior.
- Added ASP.NET Core Identity persistence in `RandWiseDbContext` with `RandWiseIdentityUser` linked to Domain `AppUser.IdentityUserId`.
- Register now creates an Identity user and linked Domain `AppUser`, then returns a JWT access token plus refresh token.
- Login validates the Identity password and issues tokens only after resolving the server-side linked `AppUser`.
- Refresh hashes the presented refresh token, resolves the active stored hash, loads the linked user server-side, revokes the used token, and stores a replacement token.
- Logout revokes the supplied active refresh token.
- `/auth/me` requires bearer auth and returns `MeResponse` from the authenticated principal's linked `AppUser`.
- JWT access-token `sub`, `NameIdentifier`, and `app_user_id` claims are derived from the Domain `AppUser.Id`.
- Refresh tokens are generated as high-entropy random values and only persisted as `sha256:` hashes.
- Added an additive EF Core migration for ASP.NET Core Identity tables and the `RefreshTokens` table.
- Added integration coverage for register/login/me/refresh/logout, token rotation, stale refresh-token reuse, and raw refresh-token non-persistence.

## API or schema decisions
- Password reset endpoints remain 501 because this task was scoped to register/login/refresh/logout/me persistence.
- JWT config lives under `Jwt` with non-secret placeholders in appsettings. Real environments must override `Jwt:SigningKey` outside source control.
- Access tokens are signed with HMAC SHA-256 and include `sub`, `ClaimTypes.NameIdentifier`, and `app_user_id` as the Domain `AppUser.Id`.
- Identity user IDs are generated as compact GUID strings and stored in `AppUsers.IdentityUserId`.
- `RefreshTokens.UserId` references `AppUsers.Id` with restrictive delete behavior.
- `RefreshTokens.TokenHash` is unique and stores only the SHA-256 hash of the high-entropy refresh token.
- Refresh-token expiry/revocation checks are performed after lookup by unique hash to avoid SQLite `DateTimeOffset` translation limitations.

## Known limitations
- Request password reset and reset password are still contract placeholders.
- No auth rate limiting or audit logging was added in this bounded persistence task.
- Production secret management is not implemented; only config keys/placeholders are present.
- The API does not automatically apply migrations at startup.

## Risks
- `src/RandWise.Api/RandWise.Api.csproj` needed a JWT bearer package reference even though it was not in the original owned-path list.
- Current refresh-token rotation is transaction-backed, but very high concurrency on SQLite may still surface provider-level write contention.
- Identity email confirmation, lockout policy tuning, and password reset flows still need product/security decisions.
