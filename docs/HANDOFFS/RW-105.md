## API or schema decisions
- Domain remains free of EF Core attributes and EF package references; all mapping is fluent in Infrastructure.
- IDs are string/TEXT primary keys with no database generation.
- Money columns are integer `long` values mapped to SQLite `INTEGER`.
- Enums are stored as numeric `INTEGER` columns to match the database spec.
- Date-only values and UTC timestamps are mapped to SQLite `TEXT` under EF Core SQLite conventions.
- Required unique constraints were configured for `AppUsers.IdentityUserId`, `FinancialProfiles.UserId`, `BudgetPeriods(UserId, StartDate, EndDate)`, `CategoryBudgets(BudgetPeriodId, CategoryId)`, `WhatsAppContacts.PhoneNumberHash`, `IncomingMessages.WhatsAppMessageId`, and `MessageInterpretations.IncomingMessageId`.
- User-owned tables include `UserId` columns needed by later authorization-filtered queries.
- Foreign keys use restrictive delete behavior to avoid accidental cascading deletion before account deletion/anonymization workflows are designed.
- API persistence registration was not wired into `RandWise.Api` because the API project currently has no Infrastructure project reference and the task ownership did not include `src/RandWise.Api/RandWise.Api.csproj`.

## Known limitations
- No repositories or transaction/application use cases were implemented.
- ASP.NET Core Identity persistence tables are not included yet; `AppUsers.IdentityUserId` is prepared to link to the future Identity user record.
- API startup does not yet call `AddRandWisePersistence` because adding the required API-to-Infrastructure reference is left for the Integration/Identity wiring task.
- System category seed data was not added; later agents should seed categories intentionally when category workflows are implemented.

## Risks
- The first Integration/Identity task that wires Infrastructure into the API must own the API project reference and startup registration to avoid crossing current ownership boundaries.
- Future migrations may need additive indexes once query patterns are known, especially dashboard and reporting queries.
- Restrictive delete behavior means deletion/anonymization workflows must explicitly order deletes or anonymize dependent rows.

## Required follow-up
- Integration or Identity agent should add the API project reference to Infrastructure and call `AddRandWisePersistence` from `Program.cs`.
- Identity agent should add Identity tables and implement AppUser creation linkage.
- Transaction, Budgeting, and WhatsApp agents should use this DbContext for persistence and preserve authenticated `UserId` filtering in all user-owned queries.
