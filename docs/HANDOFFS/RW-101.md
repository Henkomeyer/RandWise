## API or schema decisions
- Domain IDs are strings to match the TEXT primary keys in the database spec.
- Monetary values are represented as `long` integer cents. No `float`, `double` or `decimal` money types were added.
- Domain entities use `DateOnly` for date-only concepts and `DateTime` guarded to `DateTimeKind.Utc` for instants.
- No EF Core attributes, ASP.NET Core types, provider clients, schema migrations or persistence configuration were added.
- `StartingBalanceCents`, `OpeningBalanceCents` and rollover amounts allow signed balances. Budget allocations, income, safety buffer, savings commitments and transaction amounts must be non-negative or positive as appropriate.
- Transaction amount is positive; income/expense direction is represented by `TransactionType`.
- Medium-confidence restore behavior returns transactions to `NeedsReview` for 7000-8999 basis points and `Confirmed` otherwise.

## Known limitations
- Domain invariants are intentionally foundational and do not implement cross-aggregate uniqueness such as one profile per user or one category budget per category per period; those belong in persistence/application workflows.
- No safe-to-spend calculation service was added in this task; only budget-period and money/date primitives needed by later tasks are present.
- No public JSON enum string mapping was added because Domain remains transport-independent.

## Risks
