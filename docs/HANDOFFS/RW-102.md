## API or schema decisions
- Auth routes are mapped exactly under `/api/v1/auth/*` per `docs/API_CONTRACT.md`.
- Auth stubs return Problem Details with HTTP 501 where Identity/database persistence is required.
- `GET /api/v1/auth/me` returns 401 without an authenticated principal and does not bind any client-supplied user ID.
- No schema, EF migration or Identity persistence changes were made.
- Refresh tokens are generated as 64 random bytes encoded as base64url text.
- Refresh token hashes use a `sha256:` prefix for algorithm agility and store only a one-way hash. This is acceptable for high-entropy refresh tokens; passwords must still use Identity password hashing.
- Added `Microsoft.AspNetCore.Mvc.Testing` to integration tests to exercise minimal API routes through the ASP.NET Core test host.

## Known limitations
- Registration, login, refresh, logout and password reset are endpoint-shape stubs only.
- JWT issuing is intentionally disabled until a real signing key source and Identity user persistence exist.
- Refresh token store is interface-only until persistence is implemented.
- No authorization policies or rate-limit policies are wired yet.

## Risks
- The exact auth DTO fields may need additive changes when ASP.NET Core Identity and profile creation are connected.
- Later persistence work must ensure refresh token rotation is atomic and revoked tokens cannot be reused.
- Later logging work must keep passwords, access tokens and refresh tokens out of structured logs.

- Add auth rate limiting and audit events for register/login/logout/reset flows.
