## API or schema decisions
- Public transaction enum values are stable strings: `expense`, `income`, `web`, `whatsapp`, `recurring`, `import`, `confirmed`, `needsReview`, `deleted`.
- Monetary values remain `long` integer cents. No float/double money fields were introduced.
- Transaction dates use `DateOnly` and serialize as ISO date-only values under ASP.NET Core defaults.
- Pagination uses `items`, `page`, `pageSize`, `totalCount` and `totalPages`.
- Category lookup permits active system categories or active categories owned by the authenticated user.
- If a create request omits category ID, the service creates/uses an uncategorised system category for the first vertical slice.
- Endpoint module was left standalone; `Program.cs` wiring is left to Integration.

## Known limitations
- Endpoint integration depends on Integration wiring because RW-201 intentionally does not wire `MapTransactionEndpoints()` in `Program.cs`.
- Categorise changes only the transaction category. Learning user-specific category rules is deferred to later intelligence tasks.
- WhatsApp processing is out of scope and should call these same create/update/categorise use cases later.

## Risks
- The uncategorised default system category is created by the transaction service as a first-slice fallback; later category seed data may replace this with an intentional seed.
- Search currently uses provider-translated `Contains` over description, merchant and notes; later reporting/search work may need normalized search terms or indexes.

## Required follow-up
- Integration Agent should wire `api.MapTransactionEndpoints()` in `Program.cs`.
- Integration/Category agents should align system category seed IDs with the fallback uncategorised category.
- WhatsApp processing should use `ITransactionService.CreateAsync` rather than writing transactions directly.
